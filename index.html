§1 — Проблема: платформенные потоки

~1 MB
стек каждого потока (зарезервировано ОС)
~1 ms
время создания потока
~10K
практический предел потоков на JVM

Thread-per-Request

  • Классическая модель серверов
  • 1 запрос = 1 OS thread
  • При 10K RPS нужно 10K потоков
  • → OOM или деградация

Попытки решения

  • Callback Hell (Async IO)
  • Reactive Programming
  • Сложный код, трудная отладка
  • → разрыв со стандартным Java

Little's Law

  • L = λ · W
  • При W=100ms, λ=1000 RPS
  • → L = 100 потоков занято
  • Больше потоков = больше масштаб

§1 — Демонстрация предела потоков


// Сколько platform threads может создать JVM?
try {
    List<Thread> threads = new ArrayList<>();
    for (int i = 0; i < 10_000; i++) {
        Thread t = new Thread(() -> {
            try {
                Thread.sleep(10_000); // каждый поток ждёт 10 сек
            } catch (InterruptedException e) {}
        });
        t.start();
        threads.add(t);
    }
    System.out.println("Создано потоков: " + threads.size());
} catch (OutOfMemoryError e) {
    System.err.println("OOM при создании потока: " + e.getMessage());
}

// С Virtual Threads — без проблем:
for (int i = 0; i < 1_000_000; i++) {
    Thread.ofVirtual().start(() -> {
        try {
            Thread.sleep(10_000);
        } catch (InterruptedException e) {}
    });
}
System.out.println("1 миллион VT запущен успешно!");
Virtual Threads: стек в heap, начинается с ~200–300 байт. Миллион потоков = ~200 MB heap.

§3 — Lifecycle: Mount / Unmount

При блокирующей операции carrier thread освобождается и берёт другой виртуальный поток. Стек сохраняется в heap как Continuation.

§3 — Continuation: код и поведение


// Что происходит при вызове блокирующего метода в VT:
void handleRequest(HttpRequest req) {
    // 1. VT mounted на carrier thread
    String data = fetchFromDB(req.id());   // ← блокирующий вызов
    // ↑ JVM: unmount VT, сохранить стек в heap, carrier thread свободен
    // ... carrier thread выполняет другие VT ...
    // I/O завершился: JVM remount этот VT на (возможно другой) carrier
    // 2. VT продолжает выполнение со следующей строки
    process(data);
}

// Планировщик — ForkJoinPool (work-stealing):
// По умолчанию размер = Runtime.getRuntime().availableProcessors()
// Можно настроить:
System.setProperty("jdk.virtualThreadScheduler.parallelism", "8");
System.setProperty("jdk.virtualThreadScheduler.maxPoolSize", "256");

// Continuation — внутренняя JVM-абстракция:
// публичного API нет, но идея: сохраняем точку возврата + локальные переменные
// при unmount, восстанавливаем при remount

§4 — Четыре способа создать Virtual Thread


// 1. Thread.ofVirtual() — прямое создание
Thread vt1 = Thread.ofVirtual().start(() -> task());

// 2. Thread.startVirtualThread() — shortcut
Thread vt2 = Thread.startVirtualThread(() -> task());

// 3. ExecutorService (рекомендован для production)
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    Future<String> f = executor.submit(() -> callService());
    String result = f.get();
}

// 4. StructuredTaskScope (Java 21, для связанных задач)
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    var t1 = scope.fork(() -> fetchUser(id));
    var t2 = scope.fork(() -> fetchOrders(id));
    scope.join().throwIfFailed();
    return new UserWithOrders(t1.get(), t2.get());
}

// ⚠️ АНТИПАТТЕРН: НЕ оборачивать VT в пул!
// Executors.newFixedThreadPool(100) с VT внутри — бессмысленно

Читаемые stack traces и thread dumps. Без имени: VirtualThread[#42]/runnable

§5 — Pinning: виртуальный поток застрял

§5 — Pinning: диагностика и исправление

Антипаттерн (вызывает pinning):


// ⚠️ synchronized + blocking = PINNING
synchronized (this) {
    // VT пришпилен к carrier!
    String data = db.query(sql);  // блокирует carrier
    cache.put(key, data);
}

Решение — ReentrantLock:


// ✅ ReentrantLock: carrier освобождается
private final ReentrantLock lock =
    new ReentrantLock();

lock.lock();
try {
    String data = db.query(sql); // OK!
    cache.put(key, data);
} finally {
    lock.unlock();
}
      

# Диагностика pinning:
java -Djdk.tracePinnedThreads=full -jar app.jar

# Вывод при pinning:

#   ...
#   com.example.OrderService.processOrder(OrderService.java:42)  <-- PINNED HERE

§6 — ThreadLocal vs ScopedValue

§6 — ScopedValue: пример использования


// Объявление — статическая константа
static final ScopedValue<String>  REQUEST_ID = ScopedValue.newInstance();
static final ScopedValue<UserCtx> USER_CTX   = ScopedValue.newInstance();

// Установка значения и запуск кода в scope
void handleRequest(String reqId, User user) throws Exception {
    ScopedValue.where(REQUEST_ID, reqId)
               .where(USER_CTX, new UserCtx(user))
               .run(() -> {
                   // Внутри этой лямбды и всех дочерних VT:
                   processOrder();
                   logAction();
               });
    // После выхода — значения автоматически уничтожены
}

// Чтение значения (из любого дочернего потока)
void processOrder() {
    String reqId = REQUEST_ID.get(); // всегда безопасно внутри scope
    log.info("[{}] Processing order", reqId);
}

// ⚠️ Попытка читать вне scope → NoSuchElementException
// REQUEST_ID.get() // throws if not bound

§7 — Structured Concurrency

ShutdownOnFailure

Первый упавший → отменяет всех. join().throwIfFailed()

ShutdownOnSuccess

Первый успешный → отменяет остальных. Race pattern.

§7 — Structured Concurrency: код


// ShutdownOnFailure: все или никто
AggregatedResult processOrder(Order order)
        throws InterruptedException, ExecutionException {
    try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
        // fork() запускает каждую задачу в отдельном VT
        var inventory = scope.fork(() -> checkInventory(order));
        var credit    = scope.fork(() -> checkCredit(order));
        var rate      = scope.fork(() -> getExchangeRate(order.currency()));

        scope.join()           // ждём завершения всех (или первой ошибки)
             .throwIfFailed(); // пробрасываем исключение если было

        // Здесь все задачи успешно завершились
        return new AggregatedResult(
            inventory.get(), credit.get(), rate.get());
    } // auto-close: отменяет незавершённые задачи
}

// ShutdownOnSuccess: race — берём первый ответ
String fetchFromAnyReplica(List<String> replicas)
        throws InterruptedException, ExecutionException {
    try (var scope = new StructuredTaskScope.ShutdownOnSuccess<String>()) {
        replicas.forEach(r -> scope.fork(() -> httpGet(r)));
        scope.join();
        return scope.result(); // первый успешный ответ
    }
}

§8 — Blocking IO: Platform vs Virtual Threads

Virtual Threads: такая же утилизация CPU как Reactive, но с простым синхронным кодом.

§8 — Benchmark: пропускная способность

VT дают огромный прирост только для I/O-bound задач. CPU-bound: используйте ForkJoinPool/parallelStream.

§8 — Когда применять Virtual Threads?

Идеально подходит

  • HTTP-запросы к внешним API
  • JDBC / database queries
  • File I/O операции
  • gRPC / message queue consumers
  • Высококонкурентные серверы

→ I/O-bound workloads

Не поможет / вредит

  • Интенсивные вычисления (ML, crypto)
  • Тяжёлая обработка изображений/видео
  • Уже используете Reactive + всё ок
  • Код с обильными synchronized блоками

→ CPU-bound workloads

Spring Boot 3.2+

  • spring.threads.virtual.enabled=true
  • Tomcat/Jetty автоматически используют VT
  • @Async методы → VT
  • Нет изменений в бизнес-коде

→ Одна строка конфига!

§9 — Чеклист миграции

  • ✓ Заменить Executors.newFixedThreadPool(N) → Executors.newVirtualThreadPerTaskExecutor()
  • ✓ Найти все synchronized + блокирующий вызов внутри → заменить на ReentrantLock
  • ✓ Оценить использование ThreadLocal — если >1000 потоков, рассмотреть ScopedValue
  • ✓ Запустить приложение с флагом -Djdk.tracePinnedThreads=full и устранить pinning
  • ✓ Провести нагрузочный тест (k6 / Gatling) — убедиться в росте throughput
  • ✓ Записать JFR-профиль и проверить на наличие jdk.VirtualThreadPinned events
  • ! НЕ создавать пул виртуальных потоков — newFixedThreadPool с VT внутри бессмысленен
  • ! Проверить библиотеки на наличие внутренних synchronized блоков (JDBC-драйверы, Redis-клиенты)

§10 — Отладка: JFR и JVM-флаги


# 1. Включить трассировку pinning при старте
java -Djdk.tracePinnedThreads=full \
     -Djdk.virtualThreadScheduler.parallelism=8 \
     -jar app.jar

# 2. Запись JFR профиля (5 минут)
java -XX:StartFlightRecording=duration=5m,\
       filename=vthread-profile.jfr,\
       settings=profile \
     -jar app.jar

# 3. Анализ JFR (JDK Mission Control или jfr tool):
jfr print --events jdk.VirtualThreadPinned vthread-profile.jfr
jfr print --events jdk.VirtualThreadSubmitFailed vthread-profile.jfr

# 4. Thread dump включает Virtual Threads (Java 21):
jcmd <PID> Thread.dump_to_file -format=json threads.json

# 5. Полезные JVM флаги:
# -Djdk.virtualThreadScheduler.parallelism=N   (default: CPU cores)
# -Djdk.virtualThreadScheduler.maxPoolSize=N   (default: 256)
# -Djdk.tracePinnedThreads=short|full

Практическое задание: агрегатор заказов


static final ScopedValue<String> REQUEST_ID = ScopedValue.newInstance();

AggregatedResult processOrder(String requestId, Order order)
        throws InterruptedException, ExecutionException {
    return ScopedValue.where(REQUEST_ID, requestId).call(() -> {
        try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
            // Параллельно запускаем три задачи:
            var inventoryTask = scope.fork(() -> checkInventory(order));  // ~50ms
            var creditTask    = scope.fork(() -> checkCredit(order));     // ~80ms
            var rateTask      = scope.fork(() -> getExchangeRate(        // ~30ms
                                    order.currency()));
            scope.join().throwIfFailed();
            // Итог: ~80ms (max), не 160ms (sum)!
            return new AggregatedResult(
                inventoryTask.get(), creditTask.get(), rateTask.get());
        }
    });
}

Требования

  • StructuredTaskScope.ShutdownOnFailure
  • ScopedValue (не ThreadLocal)
  • ReentrantLock вместо synchronized
  • JFR без VirtualThreadPinned

Критерии оценки

  • Нет pinning events в JFR
  • Время ≈ max(50,80,30) = ~80ms
  • ScopedValue propagates в child VT
  • Корректная обработка ошибок

Итоги урока

🧵

Virtual Threads

JVM-потоки поверх carrier threads. Стек в heap. Миллионы потоков — без проблем.

⚙️

Mount / Unmount

При I/O — carrier thread освобождается. Continuation сохраняет стек в heap.

⚠️

Pinning

synchronized + I/O = беда. Заменять на ReentrantLock. Диагностика: JFR + tracePinnedThreads.

🔒

ScopedValue

Замена ThreadLocal для миллионов VT. Immutable, bounded, auto-propagated.

🏗️

Structured Concurrency

Дочерние задачи не переживают scope. ShutdownOnFailure / ShutdownOnSuccess.

📈

Когда применять

I/O-bound: огромный прирост. CPU-bound: без изменений. Spring Boot: одна строка.

Virtual Threads = простота синхронного кода + масштабируемость асинхронного подхода
